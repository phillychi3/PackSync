# PackSync 多人旅遊管理系統

## UI／UX 產品設計規格書

---

## 0. 現行視覺基準（實作時必讀）

本章定義 PackSync 已確立的視覺語言，是後續所有頁面的最高優先級設計基準。若本文件其他段落只有概念描述，應以本章的精確數值與元件配方為準。

目標不是製作一般的圓角 SaaS 介面，而是建立一個冷靜、精準、具有旅行控制中心氣質的介面。畫面由大面積冷白底、炭黑設備面板、螢光黃綠操作色、細格線、等寬系統標籤及不對稱排版組成。裝飾必須服務資訊層級，不能變成霓虹科幻或遊戲 HUD。

### 0.1 品牌名稱與文字語氣

產品正式名稱一律寫作 **PackSync**，全大寫識別字寫作 **PACKSYNC**。不要使用 PaskSync。

中文標題簡短、直接、有節奏，建議使用兩段式句型：

```text
一起出發
保持同步。

把混亂留在
出發之前。
```

英文只作為分類、狀態與設備標籤，不取代主要中文資訊：

```text
01 / TRAVEL TOGETHER
SYNC / ONLINE
TRIP / TPE-NRT-042
DAY 02 / 05
```

系統文字使用大寫英文、斜線與兩位數編號。中文不可刻意增加字距。

### 0.2 精確色彩 Token

以下色碼應直接使用，不要自行改成相近的藍、紫或高飽和霓虹色。

| Token            | 色碼      | 用途                          |
| ---------------- | --------- | ----------------------------- |
| `canvas`         | `#f4f5f2` | Landing 主背景、冷白畫布      |
| `canvas-auth`    | `#f1f2ee` | 登入與註冊主背景              |
| `surface-muted`  | `#eef0eb` | 大型次要區段                  |
| `surface-card`   | `#f7f8f5` | 淺色資訊卡片                  |
| `surface-raised` | `#e9ebe5` | 深色面板上的浮起資訊卡        |
| `ink`            | `#151817` | 主要文字                      |
| `ink-panel`      | `#171a19` | 主要深色按鈕與頁尾            |
| `panel`          | `#202423` | Landing 右側深色展示區        |
| `panel-raised`   | `#292e2c` | 深色區域內的浮動卡片          |
| `auth-panel`     | `#1c201f` | Auth 左側品牌面板             |
| `signal`         | `#d8ff36` | 唯一主要強調色、CTA、目前狀態 |
| `signal-hover`   | `#c8ef28` | 強調色 hover                  |
| `signal-dark`    | `#799800` | 淺底上的綠色圖示或狀態        |
| `status-online`  | `#8fb400` | 線上狀態燈                    |

透明度規則：

- 主要文字：`ink` 100%。
- 正文與說明：黑色 50–60%。
- 系統標籤：黑色或白色 35–45%。
- 一般分隔線：黑色 10–20%，深色底使用白色 10–15%。
- 淡化的大標題可使用黑色 28%，不可改用彩色漸層。

強調色 `signal` 只用於主要 CTA、活動狀態、關鍵數值與小型狀態燈。單一畫面中螢光色面積通常不得超過 10%。

### 0.3 字體與排版尺度

全站主要字體使用 `Inter Variable`，由 `@fontsource-variable/inter` 載入。技術資料使用系統等寬字體 `font-mono`。禁止引入裝飾性科幻字體。

核心排版配方：

| 類型          | 建議 Tailwind                                                                   | 說明                         |
| ------------- | ------------------------------------------------------------------------------- | ---------------------------- |
| Landing Hero  | `text-[clamp(3.4rem,8vw,7.5rem)] font-black leading-[0.87] tracking-[-0.075em]` | 極緊密、大膽、不換成一般粗體 |
| 大型區段標題  | `text-4xl sm:text-6xl font-black tracking-[-0.05em]`                            | 允許兩行短句                 |
| Auth 標題     | `text-4xl sm:text-5xl font-black tracking-[-0.05em]`                            | 例如「歡迎回來。」           |
| Auth 品牌宣言 | `text-5xl xl:text-7xl font-black leading-[0.95] tracking-[-0.06em]`             | 僅桌面左側面板               |
| 正文          | `text-lg leading-8 text-black/50`                                               | 最大寬度約 28–36rem          |
| 系統標籤      | `font-mono text-[10px] tracking-[0.15em]`                                       | 使用大寫英文                 |
| 分類標籤      | `font-mono text-xs font-bold tracking-widest`                                   | 前方可有編號                 |

大標題負責情緒，系統標籤負責秩序。不要把每一段文字都變成粗體或等寬字。

### 0.4 幾何、邊框與陰影

- 主要頁面使用直角或近乎直角。核心 `Button`、`Card`、`Input` 需以 `rounded-none` 覆蓋 shadcn 預設圓角。
- 邊框是主要分層手段：使用 `border-black/10`、`border-black/15`、`border-black/20`。
- 模組列表採相鄰共用邊線，外框可用 `border-t border-l`，子卡使用 `border-r border-b`。
- 淺色主要介面不使用大型柔和陰影。一般卡片使用 `shadow-none ring-0`。
- 陰影只允許出現在深色展示面板中的浮動設備卡，用來表達前後層次。
- 不使用玻璃擬態、彩色光暈、大面積漸層、膠囊卡片或全部元素都圓角的設計。
- 裁切角與條碼可作少量裝飾，但現行基準優先使用乾淨矩形與細格線。

### 0.5 shadcn-svelte 使用原則

所有互動元件優先使用 `src/lib/components/ui` 內的 shadcn-svelte 元件，再以 Tailwind `class` 客製。不要為單一頁面重新製作原生 Button、Input、Card、Dialog、Select 或 Tabs。

推薦 import：

```svelte
<script lang="ts">
	import { Button } from '$lib/components/ui/button'
	import { Input } from '$lib/components/ui/input'
	import { Card, CardContent } from '$lib/components/ui/card'
</script>
```

客製應在使用端疊加 class，不直接修改 shadcn 共用元件原始碼，除非是全產品一致的基礎 token 變更。

主要按鈕配方：

```svelte
<Button
	href="/register"
	class="group h-13 justify-between rounded-none bg-[#d8ff36] px-6 font-bold text-black ring-1 ring-black hover:bg-[#c8ef28]"
>
	建立第一趟旅程
	<ArrowRight class="size-5 transition group-hover:translate-x-1" />
</Button>
```

深色按鈕配方：

```svelte
<Button
	class="h-14 w-full justify-between rounded-none bg-[#171a19] px-5 font-bold text-white hover:bg-black"
>
	登入工作區
	<ArrowRight class="size-5 text-[#d8ff36]" />
</Button>
```

次要按鈕配方：

```svelte
<Button
	variant="outline"
	class="h-13 rounded-none border-black/25 bg-transparent px-6 font-bold shadow-none hover:bg-white"
>
	探索功能
</Button>
```

輸入框配方：

```svelte
<Input
	class="h-13 rounded-none border-black/20 bg-white px-4 text-base shadow-none placeholder:text-black/25 focus-visible:border-black focus-visible:ring-[#d8ff36]"
/>
```

模組卡片配方：

```svelte
<Card
	class="rounded-none border-0 border-r border-b border-black/20 bg-[#f7f8f5] py-0 shadow-none ring-0 hover:bg-white"
>
	<CardContent class="p-6 sm:p-8">
		<!-- 編號、圖示、標題與說明 -->
	</CardContent>
</Card>
```

### 0.6 Landing Page 標準版型

Landing Page 必須依序包含：頂部導覽、Hero、功能模組、深色結尾 CTA、深色頁尾。

#### 頂部導覽

- 高度 `h-18`，背景 `#f4f5f2` 搭配輕微 backdrop blur。
- 最大內容寬度 `max-w-7xl`，左右 padding 手機 `px-5`、桌面 `sm:px-8`。
- Logo 是 32px 炭黑方塊，內含黃綠色 `PS`，右側為粗體 `PACKSYNC`。
- 右側為「登入」ghost button 與炭黑色「免費開始」button。
- 手機可隱藏登入按鈕，但保留主要 CTA。

#### Hero

- 桌面是約 `1.08fr / 0.92fr` 的非對稱雙欄；手機只顯示左側內容。
- 最小高度為視窗高度扣除導覽列：`min-h-[calc(100vh-4.5rem)]`。
- 左欄順序：線上狀態標籤、`01 / TRAVEL TOGETHER`、巨大兩行標題、簡介、雙 CTA、能力標籤。
- 主要標題第一行使用實色，第二行使用 `text-black/28`。
- CTA 在手機垂直排列、`sm` 以上水平排列。
- 桌面右欄使用 `#202423` 深色底與 48px 細格線；不能放通用 stock photo。
- 右欄放置兩張錯位浮動資訊卡：深色「下一個行程」卡與淺色「團體狀態」卡。資訊要像真實產品資料，而不是純裝飾。

Hero 結構比例：左欄 padding 約 `px-5 sm:px-10 lg:px-14`，垂直 padding `py-20 lg:py-28`。右側展示區只在 `lg` 以上顯示，避免手機資訊過密。

#### 功能模組

- 區段背景 `#eef0eb`，上方以編號標籤與大型中文標題開場。
- 功能卡為桌面兩欄、手機單欄，共四張：行程、費用、行李、AI 助理。
- 卡片上半部保留大量空白，編號在左、Lucide 線性圖示在右。
- 卡片 hover 僅從 `#f7f8f5` 變為白色，圖示可轉深綠；不要浮起或放大。

#### 結尾 CTA 與頁尾

- 使用完整炭黑背景 `#171a19`，白色巨型短句搭配黃綠主要按鈕。
- 桌面採雙欄：左側宣言、右側三項利益與按鈕。
- 頁尾延續炭黑背景，以 10px 等寬低對比文字顯示版權與品牌句。

### 0.7 Login／Register 標準版型

Login 與 Register 必須共用同一個 AuthPanel 類型元件，只由 `mode` 控制標題、欄位及連結，避免兩頁視覺分叉。

#### 桌面

- 全螢幕雙欄，比例約 `0.88fr / 1.12fr`。
- 左欄為 `#1c201f` 品牌面板，含 52px 細格線、Logo、狀態標籤、大型品牌宣言、說明及底部三欄系統標籤。
- 左欄內容垂直分散排列，宣言落在中下區域，不要全部置中。
- 右欄背景 `#f1f2ee`，頂部有 72px 高的細邊框 header，右側顯示登入／註冊切換。
- 表單最大寬度 `max-w-md`，在右欄水平與垂直置中。

#### 手機

- 隱藏左側品牌面板，只保留右側表單。
- Header 左側顯示返回首頁，右側顯示登入／註冊切換。
- 頁面左右 padding 為 20px，欄位與按鈕維持至少 52–56px 高。

#### 表單順序與狀態

- 小型編號標籤。
- 4xl–5xl 粗體標題與一句輔助說明。
- Register 額外顯示名稱欄位。
- Email、Password，密碼尾端使用 shadcn ghost icon button 切換可見性。
- 錯誤訊息使用淡紅底與左側 4px 紅線，含 `AUTH / ERROR` 系統標籤。
- Submit 使用滿寬炭黑按鈕，右側箭頭為 `#d8ff36`。
- 送出中降低 opacity、禁用按鈕並以小型 spinner 取代箭頭，不改變按鈕尺寸。
- 底部以鎖頭圖示與 `SECURE SESSION / HTTP-ONLY` 作為低對比安全標籤。

### 0.8 圖示與內容密度

- 使用 `@lucide/svelte`，標準內容圖示 20–24px，系統標籤圖示 14–16px。
- 圖示維持預設線性筆畫，不放進彩色圓形底座。
- 頭像是唯一可使用完整圓形的常見元素。
- 每個大區塊最多出現一個視覺焦點；系統小字、格線與編號總裝飾面積不超過 15%。
- 展示資料應包含具體時間、地名、金額與狀態，例如 `14:20—16:00`、`TOKYO / TAITO`、`NT$ 860`，藉此建立產品真實感。

### 0.9 響應式重建檢查表

- 360px：無水平捲動；Hero 標題不被裁切；CTA 垂直排列；Auth 左欄隱藏。
- 390–430px：表單按鈕與輸入框高度至少 52px；頁面左右留白 20px。
- 768px：功能卡進入兩欄；主要內容留白增加。
- 1024px：Landing Hero 與 Auth 切換雙欄，深色展示面板出現。
- 1280px 以上：內容仍限制在 `max-w-7xl`，不可無限制拉寬；Auth 品牌字可提升至 7xl。

### 0.10 禁止偏離事項

- 不得將主識別色改成藍色或紫色。
- 不得使用大面積漸層、玻璃擬態、霓虹發光或頻繁 glitch。
- 不得把卡片全部改成大圓角加柔和陰影。
- 不得使用 stock travel hero photo 取代右側產品資訊面板。
- 不得只做桌面版；手機必須是第一級設計情境。
- 不得使用原生互動元件繞過已有 shadcn-svelte UI library。
- 不得用顏色單獨表示狀態，必須同時顯示文字或圖示。
- 不得在 Svelte 元件加入 `<style>`；視覺一律使用 TailwindCSS class。

### 0.11 視覺驗收標準

只看頁面截圖時，應能立即辨識以下特徵：

1. 冷白畫布搭配炭黑大型面板。
2. 黃綠色只出現在關鍵操作與狀態。
3. 大型緊密粗體中文標題與小型等寬英文標籤形成強烈尺度對比。
4. 方角 shadcn 元件、細邊框、共用格線取代圓角陰影卡片。
5. 桌面使用不對稱雙欄，手機則簡化為清楚單欄。
6. 介面像旅行協作控制系統，但仍保持一般使用者能直覺操作。

若上述六點有兩點以上不成立，即使功能正確，也不視為符合 PackSync 現行設計。

## 1. 設計目標

PackSync 採用行動裝置優先設計，核心目標是讓多人旅遊成員能快速掌握行程、費用、行李準備進度，以及透過 AI 查詢旅程相關資訊。

整體介面應具備以下特性：

- 資訊層級清楚
- 單手操作友善
- 關鍵資訊可快速瀏覽
- 多人協作狀態即時可見
- 弱網路或離線狀態下仍可查看重要資料
- 高風險資訊，例如護照與交通時間，必須具有明確提醒

## 2. 資訊架構

### 2.1 主要導覽

系統使用固定式底部導覽列，包含四個主要功能：

1. 行程
2. 費用
3. 行李
4. AI 助理

底部導覽列在主要頁面中固定顯示，進入編輯頁、詳細頁或全螢幕流程時可暫時隱藏。

### 2.2 全域頁面

除四個主要模組外，系統需提供：

- 旅程選擇與切換
- 成員管理
- 通知中心
- 個人設定
- 旅程設定
- 離線狀態提示
- 即時同步狀態提示

## 3. 全域設計系統

### 3.1 視覺風格

整體設計採用以《明日方舟》視覺氛圍為參考的 **Futuristic Minimalism（未來極簡主義）**，融合 **Barcode Aesthetic（條碼美學）** 與部分 **Cassette Futurism（卡帶未來主義）** 元素。

設計不直接複製既有遊戲介面、圖示、字體或品牌素材，而是提取其工業感、資訊分層、戰術介面與近未來設備語彙，建立 PackSync 專屬的旅遊協作視覺系統。

整體風格關鍵字：

- 未來極簡
- 戰術資訊介面
- 工業標示系統
- 高密度資訊排版
- 模組化卡片
- 條碼與序號元素
- 類比設備操作感
- 冷靜、中性、精準
- 低飽和配色
- 局部高亮識別色

---

### 3.2 視覺設計原則

#### 資訊優先

介面應以資訊閱讀效率為核心，不使用過度裝飾性的科幻效果。

每個視覺元素都應具有明確用途，例如：

- 編號代表行程順序或系統分類
- 格線協助時間與資訊對齊
- 高亮色表示目前狀態或主要操作
- 技術標籤顯示同步、定位或通知狀態

#### 模組化結構

頁面由清楚的矩形模組組成，類似工業控制面板或行動終端裝置。

模組之間使用：

- 細線分隔
- 不對稱留白
- 局部色塊
- 編號標籤
- 邊角切割
- 標題與資訊錯位排版

避免所有內容都使用相同的圓角卡片，以降低一般 SaaS 或社群應用程式的視覺感。

#### 控制裝飾密度

條碼、序號、技術文字與警示線只能作為輔助元素，不得影響主要內容閱讀。

建議裝飾元素占單一畫面的比例不超過 15%。

---

### 3.3 色彩系統

整體使用低飽和中性色作為基礎，搭配單一高辨識度強調色。

#### 基礎色

- 主背景：冷白色或極淺灰色
- 次要背景：淺灰藍或淺灰綠
- 主要文字：接近黑色的深灰
- 次要文字：中灰色
- 邊框與格線：低對比灰色
- 深色模組：炭黑或深藍灰

#### 強調色

建議選擇一種主要識別色，例如：

- 高飽和黃色
- 螢光黃綠色
- 冷青色
- 工業橘色

強調色僅用於：

- 目前選取狀態
- 主要操作按鈕
- 重要時間
- 關鍵提醒
- 活動中的系統狀態

#### 狀態色

- 成功：低飽和綠色
- 警告：工業黃色或橘色
- 錯誤：低飽和紅色
- 資訊：冷青色或藍色
- 離線：中性灰色

狀態不得只依靠顏色區分，必須同時搭配文字、圖示或狀態代碼。

---

### 3.4 字體與排版

主要內文字體使用 **Inter**，維持行動裝置上的可讀性。

技術資訊、時間、金額、編號與識別碼可搭配等寬字體，例如：

- JetBrains Mono
- IBM Plex Mono
- Roboto Mono

#### 字體使用方式

**Inter 適用於：**

- 頁面標題
- 活動名稱
- 說明文字
- 按鈕文字
- 對話內容

**等寬字體適用於：**

- 日期與時間
- 金額
- 經緯度

### 3.10 動態效果

動畫風格應快速、精準，模擬資訊載入與設備回饋。

建議效果：

- 卡片快速滑入
- 狀態燈短暫閃爍
- 線條展開
- 數字快速遞增
- 條碼掃描線
- 模組依序載入
- 文字短暫解碼後顯示

動畫時間建議控制於 120–240ms。

避免使用：

- 大幅彈跳
- 過度柔和的漂浮動畫
- 長時間光暈
- 頻繁故障或 Glitch 效果
- 影響閱讀的文字亂碼動畫

Glitch 或解碼效果僅能使用於：

- App 啟動畫面
- AI 開始處理
- 旅程載入
- 特殊狀態轉換

時間刻度建議：

- 每小時一個主要刻度
- 每 30 分鐘可顯示次要刻度

活動卡片高度依照持續時間計算。

例如：

- 30 分鐘活動：最小高度卡片
- 1 小時活動：標準高度卡片
- 2 小時活動：兩倍高度卡片

若實際比例造成卡片無法閱讀，可設定最小高度。

- 交通方式
- 預約編號
- 費用資訊
- 備註

### 時間衝突

行程重疊時：

- 使用並排卡片呈現
- 顯示時間衝突提示
- 提供快速調整時間入口

### 現在時間指示器

當使用者查看當日行程時，顯示目前時間線。

現在時間指示器需：

- 橫跨時間軸
- 顯示目前時間
- 自動更新位置

- 是否確認
- 最後確認時間

---

## 7.6 出發前確認流程

在飛機、火車、巴士或跨城市移動等重大交通事件前，系統顯示確認視窗。

確認內容包括：

- 尚未確認的重要物品
- 各成員確認狀態
- 距離出發剩餘時間

### 確認規則

每位成員需自行確認其負責物品。

其他成員不得代替確認，除非由旅程管理者執行強制確認。

### 未完成狀態

若仍有未確認物品：

- 顯示未完成數量
- 顯示負責成員
- 提供再次提醒功能
- 不應直接隱藏警告

---

# 8. AI 旅遊助理設計規格

## 8.1 AI 助理首頁

### 頁面結構

1. AI 助理標題
2. 目前旅程情境
3. 對話紀錄
4. 快速問題
5. 訊息輸入區

### 情境資訊

輸入區上方需顯示 AI 目前使用的旅程情境，例如：

- 日本東京五日遊
- 第 2 天
- 今日位置：澀谷
- 今日預算：NT$3,000

讓使用者明確知道 AI 回答是基於哪一趟旅程。

---

## 8.2 對話式查詢

使用者可詢問：

- 午餐是幾點？
- 下一個行程在哪裡？
- 今天還有多少預算？
- 誰還沒確認護照？
- 從飯店到車站需要多久？
- 幫我找附近的餐廳

### 回答格式

AI 回答應優先使用：

- 簡短文字
- 資訊卡片
- 行程卡片
- 地圖入口
- 操作按鈕

避免只提供長篇文字。

---

## 8.3 快速問題

根據目前行程，顯示可點擊的建議問題，例如：

- 下一個行程是什麼？
- 今天花了多少錢？
- 附近有什麼餐廳？
- 有哪些重要物品未確認？
- 明天天氣如何？

點擊後直接送出問題。

---

## 8.4 主動式建議

AI 可根據以下資料產生建議：

- 目前時間
- 使用者位置
- 當日行程
- 天氣
- 旅遊預算
- 交通狀態
- 餐廳營業時間

建議類型包含：

- 行程即將開始
- 建議提早出發
- 附近餐廳
- 天氣變化
- 交通延誤
- 預算過高
- 行程時間衝突

### 建議卡片

卡片內容包含：

- 建議標題
- 建議原因
- 資料來源
- 建議操作
- 忽略按鈕

例如：

> 預計 20 分鐘後下雨
> 建議將步行行程改為搭乘地鐵。

操作：

- 查看交通方式
- 更新行程
- 稍後提醒
- 忽略

---

## 8.5 AI 操作確認

若 AI 建議涉及資料修改，例如：

- 新增行程
- 修改時間
- 新增支出
- 指派行李
- 發送通知

AI 不得直接執行，必須顯示確認卡片。

確認卡片需清楚列出：

- 即將修改的內容
- 受影響成員
- 修改前後差異
- 確認與取消按鈕

---

## 8.6 AI 回答限制

當 AI 無法取得即時或完整資料時，需明確標示，例如：

- 此資訊可能不是最新狀態
- 尚未取得目前位置
- 無法存取即時交通資訊
- 目前為離線狀態

不得捏造預約資訊、價格、營業時間或交通狀態。

---

# 9. 通知設計規格

## 9.1 通知類型

系統需支援：

- 行程即將開始
- 行程時間變更
- 新增或修改支出
- 收款與付款提醒
- 行李項目指派
- 重要物品確認
- 交通事件提醒
- AI 主動建議

## 9.2 通知優先級

### 一般通知

例如：

- 新增一筆餐費
- 行李清單已更新

### 重要通知

例如：

- 行程時間已變更
- 有新的付款請求

### 緊急通知

例如：

- 護照尚未確認
- 距離登機時間只剩 30 分鐘
- 重要交通行程即將開始

不同優先級需使用不同視覺層級與推播策略。

## 9.3 通知中心

每筆通知顯示：

- 通知類型
- 標題
- 說明
- 時間
- 是否已讀
- 對應操作

通知應可直接開啟相關行程、支出或行李項目。

---

# 10. 多人協作與同步狀態

## 10.1 即時同步

當其他成員修改資料時，畫面應即時更新。

可顯示輕量提示：

> 小華剛剛更新了晚餐時間。

避免每次更新都使用阻斷式彈窗。

## 10.2 同時編輯

若兩位使用者同時編輯同一筆資料：

- 顯示目前正在編輯的成員
- 儲存前檢查資料版本
- 若內容發生衝突，顯示差異比較
- 不得直接覆蓋另一位成員的修改

## 10.3 同步狀態

需支援：

- 同步中
- 已同步
- 尚未同步
- 同步失敗
- 離線編輯

狀態通常顯示於頁面頂部或資料項目旁。

---

# 11. 離線使用規格

## 11.1 可離線查看內容

至少包含：

- 已載入的行程
- 重要地點與地址
- 預約資訊
- 行李清單
- 成員聯絡資訊
- 最近的費用紀錄

## 11.2 離線編輯

離線狀態可執行：

- 勾選行李項目
- 新增行李項目
- 新增支出
- 修改行程備註
- 建立待同步資料

所有離線修改需標記為「尚未同步」。

## 11.3 重新連線

恢復網路後：

- 自動同步離線資料
- 顯示同步進度
- 發生衝突時要求使用者處理
- 同步成功後顯示簡短提示

# 12. 頁面狀態規格

每個主要頁面都需設計以下狀態：

## 12.1 載入中

使用骨架畫面，避免只顯示全頁旋轉圖示。

## 12.2 空白狀態

空白狀態需包含：

- 清楚的狀態說明
- 引導文字
- 主要操作按鈕

例如：

> 今天還沒有行程
> 新增第一個活動，開始規劃旅程。

## 12.3 錯誤狀態

顯示：

- 錯誤原因
- 重新嘗試按鈕
- 可繼續使用的替代操作

## 12.4 權限不足

顯示使用者無法執行操作的原因，例如：

> 只有旅程管理者可以刪除此行程。

## 12.5 離線狀態

顯示目前正在使用離線資料，並標示最後同步時間。

# 16. 驗收標準

## 行程管理

- 使用者可在時間軸與地圖間切換
- 活動區塊能反映活動持續時間
- 可查看完整活動資訊
- 行程衝突時能顯示警告
- 行程更新可同步給其他成員

## 費用管理

- 支援單一與多位付款人
- 支援四種分攤方式
- 分攤結果可即時計算
- 可顯示每位成員的應收與應付
- 債務最小化結果可供成員確認

## 行李管理

- 成員可共同編輯行李清單
- 每個項目可指定負責人
- 重要物品需手動確認
- 可顯示個人與團體準備進度
- 重大交通事件前可發送提醒

## AI 助理

- 可查詢行程、預算與行李資訊
- 回答需基於目前旅程資料
- 修改資料前必須要求確認
- 無即時資料時需清楚說明限制
- 推薦內容可連結至相關操作

## PWA 與同步

- 主要資料可離線查看
- 離線修改可在恢復網路後同步
- 多人編輯結果可即時更新
- 同步錯誤與衝突具有處理介面
- 支援 Web Push API 推播通知
